# 急上昇商品セクション ロジック設計メモ

トップページ（/oliveyoung）に「急上昇商品」を追加するための定義・ロジック・表示案の整理。今回は実装せず、設計のみ記載。

---

## 1. 現在のランキングデータ構造

### 商品ランキング（oliveyoung_rankings）

| 項目 | 内容 |
|------|------|
| **コレクション** | `oliveyoung_rankings/{runDate}` ＋ サブコレクション `items` |
| **runDate** | 日次実行の日付（YYYY-MM-DD）。`getRankingRunDates()` で降順取得可能。 |
| **items** | docId = 順位（rank）。各ドキュメントに `rank`, `goodsNo`, `name`, `brand` を保存。 |
| **rankDiff / isNew** | **商品ランキングの items には現状保存されていない**。Web の型では `rankDiff?`, `isNew?` を読むが、crawler の `saveRankingItem` はこれらを渡していない。 |

### ブランドランキング（brand_rankings）参考

- `brand_rankings/{runDate}/items` には **rankDiff**（前日順位 − 当日順位、正の値＝上昇）と **isNew**（前日にランク外）が **crawler 側で計算・保存**されている。
- 急上昇ブランド・NEWランクインブランドはこのデータをそのまま表示している。

### 比較用に必要なデータ

- 「急上昇商品」を出すには **「前日（または前回）の順位」** との比較が必要。
- 現状、商品ランキングは **日付ごとのスナップショット**のみ。前日分を別ドキュメント（別 runDate）として読めば、**2日分のランキングを突き合わせて計算**できる。

---

## 2. 急上昇の定義案

### A. 順位上昇型（ランクアップ）

- **定義**: 前回 runDate にもランクインしており、今回の順位が前回より**上がっている**（数字が小さくなった）。
- **上昇幅**: `前回 rank − 今回 rank`。正の値＝上昇。例: 前回 25 位 → 今回 5 位 ⇒ +20。
- **ソート**: 上昇幅の大きい順（+20, +15, …）で並べると「急上昇」として分かりやすい。

### B. 新規ランクイン（NEW）

- **定義**: 前回 runDate のランキングには**存在しなかった**商品が、今回ランクインしたもの。
- **表示**: 「NEW」や「新規ランクイン」として区別可能。順位上昇型とまとめて「急上昇」としても、別枠にしてもよい。

### C. 安定型（補助・将来）

- **定義**: 例として「3日連続 TOP10」など。複数 runDate の履歴が必要。
- **初期**: データ取得・計算が重くなるため**見送り**。A＋B を優先。

---

## 3. 初期実装として現実的なロジック

### 比較粒度

- **「今日 vs 前日」** で十分。runDate は日次なので、`runDates[0]`（最新）と `runDates[1]`（1つ前）の2つを取得し、その2日分で比較する。
- 休日などで runDate が飛ぶ場合は「直近2回の runDate」で比較（今日 vs 前回実行日）。実装は「runDates[0] と runDates[1]」でよい。

### 既存データだけで実現するか

- **可能**。追加の Firestore フィールドや新規コレクションは不要。
- 手順:
  1. `getRankingRunDates()` で先頭 2 件（例: `runDateLatest`, `runDatePrev`）を取得。
  2. `getRankingByDate(runDateLatest)` と `getRankingByDate(runDatePrev)` で両日のランキングを取得。
  3. 前日分の `goodsNo → rank` の Map を作成。
  4. 当日の各 item について:
     - 前日 Map に goodsNo が**無い** → **新規ランクイン（B）**。
     - 前日 Map に**ある** → 前日 rank と当日 rank を比較し、当日の方が良ければ **順位上昇（A）**。`rankDiff = 前日 rank − 当日 rank`（正＝上昇）。
  5. 順位上昇したものは rankDiff の大きい順、新規ランクインは「NEW」としてリスト化。必要なら両方を混在させて「急上昇」として TOP N 件を選ぶ。

### 追加データの要否

- **不要**。上記はすべて既存の `oliveyoung_rankings/{runDate}/items` の読み取りのみで完結する。
- 将来的に「crawler で rankDiff/isNew を保存」する場合は、ランキング Job 内で前日ランキングを読み、`saveRankingItem` に `rankDiff`・`isNew` を渡す拡張で対応可能（その場合は新規 Job ではなく既存 Job の拡張）。

## 4. 表示仕様の案

| 項目 | 案 |
|------|-----|
| **形態** | 既存の「今日の注目商品」と同様の**商品カード**（画像・名前・ブランド・商品詳細リンク）が扱いやすい。 |
| **件数** | 初期は **5 件**程度。順位上昇＋新規を合わせて「急上昇商品 TOP5」など。 |
| **+順位表示** | 順位上昇した場合は **「▲+20」** のように上昇幅を表示すると分かりやすい。新規は「NEW」バッジ。 |
| **並び順** | 順位上昇は上昇幅の大きい順。新規は当日順位が良い順など。混在させる場合は「上昇幅優先」でまとめて TOP5 でも可。 |
| **データなし** | 前日ランキングが無い（runDate が1件だけ等）場合は、急上昇セクションは非表示または「準備中」でよい。 |

## 5. 実装方針（crawler / frontend / Firestore）

### 案1: Frontend で計算（推奨・初期）

| 項目 | 内容 |
|------|------|
| **どこで計算するか** | Web（Next.js）の /oliveyoung ページ、または `oliveyoung-rankings` 用の取得関数内。 |
| **データ** | 既存の `getRankingRunDates()` と `getRankingByDate(runDate)` のみ。 |
| **Firestore** | 新規保存なし。既存の読み取りのみ。 |
| **メリット** | 既存データだけで完結。crawler を触らない。ランキング本体 Job と完全分離。 |
| **デメリット** | トップ表示のたびにランキングを2日分読む（読む件数は items 数に依存）。 |

### 案2: Crawler で rankDiff / isNew を保存（将来）

| 項目 | 内容 |
|------|------|
| **どこで計算するか** | 既存のランキング実行パイプライン内。当日ランキングを保存する前に、前日ランキングを読み、各 item に rankDiff / isNew を付与してから `saveRankingItem` に渡す。 |
| **Firestore** | `oliveyoung_rankings/{runDate}/items/{rank}` に `rankDiff`, `isNew` を追加保存（merge）。 |
| **メリット** | フロントは「当日ランキングを読むだけ」で急上昇を表示できる。表示側のロジックが単純。 |
| **デメリット** | ランキング本体 Job の変更が必要。変更禁止事項で「ランキング本体と補完Jobの分離」があるため、**本体 Job の拡張**として慎重に扱う必要がある。 |

### 初期実装の推奨

- **案1（Frontend で計算）** を採用する。
- 既存データのみで実現でき、crawler に手を入れず、ランキング本体・補完Jobの分離も崩さない。
- 将来的に負荷や実装の都合で「crawler で rankDiff/isNew を保存」に切り替える場合も、表示仕様（急上昇＝順位上昇＋新規）はそのまま使える。

## 6. 初期実装案（要約）

1. **取得**: `getRankingRunDates()` で先頭2件の runDate を取得。2件無ければ急上昇セクションは出さない。
2. **計算**: 2日分の `getRankingByDate` を実行し、前日の goodsNo→rank Map を用意。当日 items に対して「新規ランクイン」と「順位上昇（rankDiff 正）」を判定。
3. **リスト化**: 順位上昇は rankDiff 降順、新規は当日 rank 昇順などで並べ、上位 N 件（例: 5件）を「急上昇商品」として採用。
4. **表示**: /oliveyoung に「急上昇商品」ブロックを1つ追加。既存の注目商品と同様のカード＋「▲+20」または「NEW」表示。商品詳細リンクは `/oliveyoung/products/{goodsNo}`。
5. **補完**: 表示用に画像・名前等が必要なら、既存の `getRankingTopNWithProducts` と同様に `getOliveYoungProductByGoodsNo` で補完する。または「急上昇用に N 件だけ商品情報を補完する関数」を1本用意する。
